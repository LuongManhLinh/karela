-Goal-
Given a text document containing Agile User Stories, technical requirements, non-functional requirements (NFRs), or system descriptions, extract a highly granular, atomic Software Engineering Knowledge Graph. This graph will be used in Neo4j to analyze architecture and detect Agile INVEST criteria defects.

{system_context}

-Extraction Rules-
1. **ATOMICITY IS MANDATORY:** The `entity_name` MUST be extremely concise (1-3 words maximum). NEVER use full sentences or clauses. 
2. **CANONICALIZATION:** Normalize synonyms. If you see "Book a ride" and "Request a vehicle", extract both as "BOOK_RIDE".
3. **ANTI-AMBIGUITY:** AVOID extracting generic personas or components. Resolve "User" to "PASSENGER" or "DRIVER" based on context.
4. **AGILE FORMAT PARSING:** * "As a [role], I want to [action], so that [value]" -> map to ROLE, ACTION, and VALUE.
   * If no VALUE found directly in the format, try to infer it from the description or acceptance criteria.
   * "Given [condition], When [event/action], Then [state/rule]" -> map to CONDITION, EVENT, ACTION, STATE, and RULE.

-Steps-
1. Entity Extraction
Extract the following information:
- entity_name: Capitalized, strictly 1-3 words, normalized.
- entity_type: MUST be one of the following:
  [KEY, ROLE, ACTION, VALUE, RESOURCE, SYSTEM, COMPONENT, DATA, RULE, CONDITION, EVENT, STATE, UI_ELEMENT, ASSUMPTION, CONCEPT, ACCEPTANCE_CRITERIA]
  * Definitions:
    - KEY: Unique identifier of the story or requirement (e.g., SCRUM-123).
    - ROLE: User persona or actor ("As a...").
    - ACTION: A single verb-noun pair ("I want to EXPORT PDF").
    - VALUE: Business value or goal ("So that...").
    - RESOURCE: Database table, file, or shared asset.
    - SYSTEM: Backend service, 3rd party API, infrastructure.
    - COMPONENT: Specific software module.
    - DATA: Specific data payloads or variables.
    - RULE: NFRs, business policies, security standards.
    - CONDITION: Prerequisites ("Given...").
    - EVENT: Temporal occurrences ("When...").
    - STATE: The status of an entity (e.g., PENDING, COMPLETED).
    - UI_ELEMENT: Buttons, modals, or frontend views.
    - ASSUMPTION: Implicit dependencies or undocumented expectations.
    - CONCEPT: Abstract business domain ideas (e.g., CANCELLATION, PRIVACY).
    - ACCEPTANCE_CRITERIA: Specific testable scenarios.
- entity_description: Comprehensive description.

Format: ("entity"<|><entity_name><|><entity_type><|><entity_description>)

2. Relationship Extraction
Identify clearly related (source_entity, target_entity) pairs. 
- relationship_description: MUST START WITH a single uppercase keyword in brackets [ ]. Use standard types:
  [TRIGGERS, REQUIRES, BLOCKS, DEPENDS_ON, MODIFIES, READS, ENFORCES, RESULTS_IN, RESTRICTS, CONFLICTS_WITH, TRANSITIONS_TO, VALIDATES, ALLOWS, PREVENTS, INITIATES, ACHIEVES, IMPLEMENTS, CONTAINS, MEASURED_BY, TESTS]
  
  * Logic Flow Requirements:
    - Link ROLEs to ACTIONs using [INITIATES].
    - Link ACTIONs to VALUEs using [ACHIEVES].
    - Use [ALLOWS] vs [PREVENTS] to map opposing rules on CONCEPT or SYSTEM entities.
    - Link ACTIONs to STATEs using [TRANSITIONS_TO].
- relationship_strength: Numeric score (1-10) indicating criticality.

Format: ("relationship"<|><source_entity><|><target_entity><|><relationship_description><|><relationship_strength>)

3. Return output in English as a single list of all the entities and relationships. Use **##** as the list delimiter.

4. When finished, output <|COMPLETE|>
