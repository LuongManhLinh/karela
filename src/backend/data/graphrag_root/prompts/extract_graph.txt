-Goal-
Given a text document containing Agile User Stories, technical requirements, non-functional requirements (NFRs), or system descriptions, identify all entities of specific types from the text and all relationships among the identified entities. This will be used to build a Software Engineering Knowledge Graph to analyze dependencies, architecture, and validate Agile INVEST criteria.

-Steps-
1. Identify all entities. For each identified entity, extract the following information:
- entity_name: Name of the entity, capitalized
- entity_type: MUST be one of the following types: 
  [KEY, EPIC, ROLE, ACTION, VALUE, RESOURCE, SYSTEM, COMPONENT, DATA, RULE]
  * Definitions:
    - KEY: Ticket identifier (e.g., SCRUM-123).
    - EPIC: The larger body of work or feature grouping.
    - ROLE: User persona, actor, or system initiator.
    - ACTION: What the user or system is doing (e.g., EXPORT PDF, LOGIN).
    - VALUE: Business value, goal, or outcome.
    - RESOURCE: UI component, database table, specific file, or shared asset.
    - SYSTEM: Backend service, 3rd party API, infrastructure.
    - COMPONENT: Specific software module, library, or package.
    - DATA: Specific data payloads, states, or variables.
    - RULE: Non-functional requirements (NFRs), business policies, constraints, thresholds, security standards (e.g., E2E ENCRYPTION, 500MS TIMEOUT, SKELETON LOADING).
- entity_description: Comprehensive description of the entity's attributes, goals, or technical nature.
Format each entity as ("entity"<|><entity_name><|><entity_type><|><entity_description>)

2. From the entities identified in step 1, identify all pairs of (source_entity, target_entity) that are *clearly related* to each other. Focus on dependencies, shared resources, value achievements, and potential conflicts.
For each pair of related entities, extract the following information:
- source_entity: name of the source entity, as identified in step 1
- target_entity: name of the target entity, as identified in step 1
- relationship_description: MUST START WITH a single, uppercase keyword in brackets [ ] categorizing the relationship type. You SHOULD use one of the following standard types if applicable: [IMPLEMENTS, BELONGS_TO, DEPENDS_ON, BLOCKS, REQUIRES, PERFORMS, ACHIEVES, AFFECTS, MODIFIES, ENFORCES, CONSTRAINED_BY, CONFLICTS_WITH, USES]. ONLY IF none of these fit, you may create a new, concise, uppercase keyword (e.g., [AUTHENTICATES]). Follow the bracketed type with a detailed explanation as to why they are related. (Example: "[DEPENDS_ON] The payment service relies on the auth system.")
- relationship_strength: a numeric score indicating the strength or criticality of the relationship (1-10)
Format each relationship as ("relationship"<|><source_entity><|><target_entity><|><relationship_description><|><relationship_strength>)

-Special Rules-
* If the text explicitly mentions EXPLICIT_DEPENDENCIES, blocks, or other specific Jira Keys, extract them as KEY entities and create [DEPENDS_ON] or [BLOCKS] relationships between the current KEY and those referenced entities.
* If an EPIC is mentioned, extract it as an EPIC entity and create a [BELONGS_TO] relationship from the current KEY to the EPIC.

3. Return output in English as a single list of all the entities and relationships identified in steps 1 and 2. Use **##** as the list delimiter.

4. When finished, output <|COMPLETE|>
